# Archival Documentation

This document describes the long‑term archival and reproducibility
infrastructure of the **Assaying Anomalies** Python library.  It
explains how the repository is organised, how to generate and
interpret research artifacts, and how to verify that the translation
faithfully reproduces the functionality of the original MATLAB
version.  The goal is to ensure that future researchers can replicate
the results without ambiguity or reliance on proprietary data.

## Repository overview

The project is organised into several top‑level directories.  The
library code lives under `aa/`, while scripts for running the
anomaly pipeline and replication experiments reside in `scripts/`.
Documentation is kept in `docs/`, and a collection of YAML
configuration files defining replication experiments is stored in
`replication/configs/`.  Synthetic data generation utilities allow
users to experiment without external data sources【579088734886326†L156-L165】.

Within the `replication/` directory you will also find subdirectories
for outputs (tables, figures and logs) and a manifest describing the
results【579088734886326†L183-L214】.  The command‑line scripts in
`scripts/` orchestrate the full replication workflow, from loading
data and computing signals to forming portfolios, running regressions
and writing tables and figures【579088734886326†L197-L214】.

## Artifact generation process

To create a reproducible research artifact, run the
`build_research_artifact.py` script.  This script first executes
`verify_full_project.py`, which checks that the package imports
properly, that synthetic data can be generated and that the test
suite passes.  It then runs a replication experiment using a
configuration from `replication/configs/` on purely synthetic data,
collects the resulting tables, figures and logs and copies the
configuration file and verification report into the artifact.  A
metadata file capturing software versions, dependency versions,
experiment configuration and a timestamp is produced in
`artifact/metadata.json`, and a manifest listing all files in the
artifact is written to `artifact/manifest.json`.

The resulting `artifact/` directory contains everything needed to
reproduce the experiments:

- `outputs/`: tables, figures and logs generated by the replication
  scripts.
- `configs/`: the YAML configuration files used to run the experiments.
- `reports/`: verification reports summarising the health of the
  repository.
- `metadata.json`: reproducibility metadata including software and
  environment versions.
- `manifest.json`: a machine‑readable manifest listing all artifact
  contents and providing traceability.

## Reproducibility guarantees

The Assaying Anomalies Python library emphasises deterministic
behaviour and strict functional parity with the original MATLAB
implementation.  The synthetic data generator seeds all random
number generators, and the replication scripts write machine‑readable
manifests and metadata to guarantee that outputs can be traced back
to inputs【579088734886326†L197-L214】.  Environment specification files
(`environment.yml` and `requirements-dev.txt`) pin the versions of
Python and all dependencies to eliminate ambiguity during
installation【579088734886326†L131-L149】.

By packaging outputs, configurations and metadata together, the
artifact ensures that results can be reproduced years into the
future.  Researchers can compare their own runs against the
verification report to confirm that their environment matches the
archived one.

## How to verify the translation project

To verify that the Python translation faithfully reproduces the
original MATLAB library, run the following command from the
repository root:

```bash
python scripts/verify_full_project.py
```

This script will import the package, generate synthetic data, run an
example pipeline, execute a replication experiment on synthetic data
and run the test suite.  It produces a JSON report summarising the
status of each check.  Any failures will cause the script to exit
with a non‑zero status and should be investigated before
archiving the project.

For long‑term preservation, use:

```bash
python scripts/build_research_artifact.py --artifact-dir artifact
```

to generate a complete artifact bundle.  The resulting `artifact`
directory can be zipped and stored alongside the codebase and
documentation.  Running `scripts/prepare_final_release.py` will build
source and wheel distributions, create a repository snapshot and
assemble release notes from the archival documentation and project
completion report.
